This page is the architectural commitment summary. It exists to make explicit what is **locked** at the CAOS planning level (cannot be casually changed without an design decisions supersession) and what is deliberately **kept open** (architectural pre-investments that preserve future optionality without committing engineering today).

The substance comes from three sources:

1. The **26 locked design decisions** in the CAOS planning corpus (`.digest/adrs/`). These are referenced in every CAOS session as the non-negotiable architectural foundation.
2. The **10 architectural pre-investments** from the file-13 finance-team expansion thesis (§7.5).
3. The **8 additional pre-investments** from the file-14 stakeholder flywheel research (§21).

Together: 26 locked decisions + 18 reversible-but-deliberate pre-investments. The first set says "this we will not change." The second set says "this we have not built, but we have left a door for it."

## The 26 locked design decisions (summary)

The full design decisions set is documented in the CAOS planning corpus (`.digest/adrs/README.md`) and is the canonical list. The summary below groups the 26 by area; the individual design decisions are referenced throughout the [product overview](/caos/product/overview/), [document intelligence pipeline](/caos/product/document-intelligence/), and [Tally / ERP integration](/caos/product/tally-erp-integration/) pages.

### Foundational (design decision to design decision)

- **design decision**: Single Flutter app with role-based mode switch (CA + Client modes). Split into two store listings only if metrics demand.
- **design decision**: Account-first OTP authentication (phone + email) for all personas. **No passwords.** No anonymous token+PIN-only entry. (Reserved for Year-2 supersession when enterprise IdP / SAML / SCIM becomes a procurement requirement.)
- **design decision**: Code-first OpenAPI 3.1 generated by Huma typed handlers. Contract tests in CI; spec drift fails the build. **OpenAPI is the contract.**
- **design decision**: Desktop web ships both CA mode AND Client portal mode (full parity with mobile).

### Architecture (design decision to design decision)

- **design decision**: React 19 + Vite 8 + Tailwind 4 + TanStack Query / Router + shadcn/ui — built fresh from first principles, do **not** port `web/caos`.
- **design decision**: Huma (huma.rocks) for HTTP — typed handlers → native OpenAPI 3.1 — over chi router.
- **design decision**: Hand-rolled pgx/v5 for DB queries. No ORM. No sqlc.
- **design decision**: Single monorepo at `/home/rakesh/work/CAOS/`.
- **design decision**: **Provider interface for LLMs** day one. LlamaIndex-heavy primary impl. Gemini / Claude / OpenRouter / local OSS all feasible via config-only later additions. Per-doctype provider routing supported. **Pipeline never imports a provider SDK directly.**
- **design decision**: **lit local + LlamaParse agentic only at start.** Defer cloud OCR (Textract / Claude vision) until volume known.
- **design decision**: **Tenant isolation: SHARED (RLS) + SCHEMA (per-firm Postgres schema, search_path)** wired day one. **DEDICATED deferred** (`firms.db_host` reserved as a column).
- **design decision**: Mapping memory scope is **per-client primary + per-firm fallback**. Promote/override UI; show source scope on every match.
- **design decision**: Pluggable rule-set engine + exhaustive scope of all accounting-impacting docs. Ship invoice rule set first; add doctype rule sets per-phase.
- **design decision**: Schema permissions are sysadmin-only at launch. Firm clone-and-override deferred to Phase 6+. Reserve `doc_type_schemas.tenant_id` column.
- **design decision**: Contact access uses role-based access bundles only at launch (DIRECTOR / PARTNER / KARTA / SELF = full; ACCOUNTANT = financial+upload; FAMILY_MEMBER = personal). Reserve `client_contacts.access_level JSONB` for future override.

### Onboarding (design decision to design decision)

- **design decision**: Demo data default-on for new CA signups with prominent DEMO badges + sandboxed view + one-click wipe.
- **design decision**: Importer at launch is **Excel/CSV only**. Tally backup + ca-tree-scanner deferred to Phase 4+.
- **design decision**: Session persistence — **CA mode 90-day refresh, Client mode 30-day refresh, Access JWT 15min, auto-rotate. No "remember me" toggle.**
- **design decision**: Branding at launch is **logo + accent color only.** Custom email-from / subdomain / domain deferred to Phase 6+.
- **design decision**: Messaging channels at launch: **Email programmatic + WhatsApp click-to-send only.** No SMS gateway. Email is the OTP delivery channel; phone OTP via SMS is Phase 4+.
- **design decision**: OTP format — email contains BOTH 6-digit code AND magic link (equal weight). 5-min token, single-use, IP-bind.
- **design decision**: First-time client UX is a brief 1-screen welcome ("{Firm}'s portal | Welcome, {Name}") then directly to the request task.
- **design decision**: Solo-CA UX is the same UI as multi-member. Members tab visible with solo CA listed alone + "Invite a teammate" CTA.

### Doc-intelligence taxonomy (design decision to design decision)

- **design decision**: Post-invoice sequencing is strict sequential by daily-grind impact: **α bank reco (Phase 4) → β GST returns (Phase 5) → γ TDS+Form16+26AS (Phase 6) → δ compliance evidence (Phase 7).** Each path depth-first, fully production-ready before moving on.
- **design decision**: **LlamaIndex-first uniform pipeline** for every extractable doctype. Schema + parsing instruction + validation rule set per doctype. **Regex stays as test infrastructure only (corpus-accuracy rig) — NOT in production routing path.**
- **design decision**: **~100 system_doc_types at launch** = finoplex's seeded ~70 + ~30 additions covering common accounting/tax/audit docs. Skip specialty (FEMA / FSSAI / drug license / IEC / regulatory licenses) until Phase 6+.

The full table of 26 design decisions and the detailed rationale per design decisions is in the CAOS planning corpus. For working purposes on the docs site, the summary above plus the cross-references in the other product pages is sufficient.

## The 10 file-13 architectural pre-investments

These are the no-regret pre-investments that keep the option open to expand from the CA wedge to the finance-team layer. They are cheap to make in Phase 0-3; not making them is expensive later (most cost a few engineering-days now vs weeks of refactoring later).

### 1. Person decoupled from firm-member and client-contact

A single `persons` table holds the global identity. The same Person can be a CA's staff at firm A AND a finance-team user at separate org B AND a director-contact for client C — without identity duplication. This is what allows the [in-house CA insight](/caos/stakeholders/composite/) (60.8% of ICAI members work in industry) to translate into product reality: the same person can hold both the CA-firm role and the in-house finance-team role.

### 2. `entity` as the canonical noun (CA UX still says "Client")

The data model treats client and group-entity as the same shape. The CA-mode UX surfaces the term "Client"; finance-team UX surfaces "Entity" / "Subsidiary." The `clients` table is preserved for backward compatibility but conceptually represents `entity`. The `entity_type` enum already covers 22 types (per finoplex `migrations/000030`); extension to add `BUSINESS_GROUP`, `FOREIGN_SUBSIDIARY`, `PROJECT_OFFICE`, `BRANCH_OFFICE` is additive.

### 3. Currency column reserved on every money-bearing table

Every money-bearing table has `currency CHAR(3) NOT NULL DEFAULT 'INR'`. Money values are stored as **INR paise (`int64`)** per the CAOS money convention. No FX revaluation logic is shipped at launch — but no schema migration is needed when a future enterprise customer requires multi-currency. The currency column reservation alone saves a 2-week migration on every money table.

### 4. Doctype catalog namespace reservations

The doctype catalog reserves namespaces but does not seed them at launch:

- `AP_*` — accounts payable doctypes (vendor invoice, vendor credit note, payment advice).
- `AR_*` — accounts receivable doctypes (customer invoice receipt, balance confirm).
- `EXP_*` — expense doctypes (expense receipt, mileage log).
- `IFC_*` — internal financial controls evidence doctypes.
- `NOTICE_*` — notice intake doctypes (GST DRC-01, IT 142(1), MCA show-cause).
- `BRSR_*` — BRSR Core taxonomy doctypes.
- `CONTRACT_*` — contract metadata doctypes.
- `FAR_*` — fixed asset register doctypes.
- `INTERCOMPANY_*` — intercompany linkage doctypes.
- `BOARD_*` — board-pack and board-resolution doctypes.

Reservation means: the namespace is documented; the seed migration leaves it empty; new doctypes added later do not collide with existing codes.

### 5. Public REST API contract drafted

`/api/v1/...` paths for read-only auth + clients/entities + documents + filings are drafted in OpenAPI. API key path (`Authorization: Bearer <api_key>`) is enabled even if not externally documented at launch. **Pulled earlier (Phase 5-6) from a much later original scoping because enterprise buyers will not adopt without integration paths.**

### 6. Webhook infrastructure

HMAC-signed event delivery framework drafted in Phase 0. Used internally for module decoupling at launch. Externally exposed in Phase 5-6 as part of the public API surface. **Same pull-forward as item 5.**

### 7. Notice intake doctype + manual triage workflow

`NOTICE_GENERIC` doctype shipped in Phase 1 inbox even though no extraction is configured. Manual triage workflow (intake → assign → due date → reply attached) reuses existing primitives **P6** + **P10**. The forward-looking benefit: the notice intake use case (UC5 in [/caos/product/use-cases/](/caos/product/use-cases/)) ships from Phase 1 in basic form, expandable to LLM-extracted notices in Phase 5+.

### 8. design decision (OTP-only) re-opened with successor planned

The design decisions remains locked at launch. A successor design decisions ("OTP for SMB + SAML / OIDC for Enterprise") is planned for Year 2 — pre-condition for the first Enterprise deal. Documenting the successor *now* (not a year from now) signals the architectural stance to procurement teams.

### 9. DEDICATED tenant model gets a working stub

The third tenant isolation mode (DEDICATED, on `firms.db_host`) is documented but not implemented. The `WithTenant` dispatch layer returns a clear "deferred" path with the architecture documented. This means a future BFSI / PSU customer who requires VPC-isolated DB can be onboarded with predictable engineering effort (not architectural surprise).

### 10. Multi-entity grouping primitive (schema-only)

A `entity_group` table (parent ↔ subsidiaries) is added to the schema. CA mode never surfaces it. Business and Enterprise modes use it for group-consolidation context. **Schema-only — no UI — at launch.** This preserves the option to ship multi-entity grouping in Phase 5+ without a destructive schema migration.

## The 8 file-14 architectural pre-investments

The file-14 stakeholder flywheel research adds 8 more pre-investments based on verified-data triggers (the AA ecosystem maturity, the Big-6 audit concentration, the BRSR mandate glide path, the eMudhra DSC market share, etc.). These extend the file-13 list.

### 11. Vendor master schema (separate from client/entity)

The vendor-network flywheel (Phase 2 of the [composite flywheel](/caos/flywheel/composite/)) cannot start without a first-class vendor entity *decoupled from "client" semantics*. A `vendors` table is added with PAN/GSTIN/MSME/IEC validators and bank-account linkage. The vendor primitive is **P2** in the [atomic primitives list](/caos/product/atomic-primitives/), but the vendor master is what makes vendor onboarding (UC4) shippable as a standalone module rather than a CA-side hack.

Maps to use cases UC3, UC4, UC8, UC22.

### 12. AA TSP / FIU integration architecture

Phase 4 fact-graph requires AA reads. The provider interface for AA is designed analogously to the LlamaIndex provider abstraction. Concrete implementations live under `internal/aa/<tsp_name>/` (CAMSFinServ, Anumati, Perfios, Setu).

The verified ecosystem context: **17 NBFC-AAs operational** ([Sahamati](https://sahamati.org.in/account-aggregators-in-india/)); **1,157 total AA participants** ([Sahamati FIP-FIU page](https://sahamati.org.in/fip-fiu-in-account-aggregators-ecosystem/)); **22.31 million AA monthly consents in March 2026** with 78.6% YoY growth ([Sahamati dashboard](https://sahamati.org.in/aa-dashboard/) / [Business Standard](https://www.business-standard.com/finance/news/aa-ecosystem-consent-processing-up-78-in-fy25-shows-sahamati-data-125031301053_1.html)). The AA framework is dramatically more mature than the file-09 research originally implied — building the architecture stub in Phase 3 is justified by the trajectory.

### 13. NFRA-compliant audit working-paper schema

The Audit Workspace bridge (Phase 3 / use case B1) ships with NFRA-2-compatible structure to make Big-6 adoption frictionless. Working-paper templates, query log structure, and evidence-locker linkage all match the schema NFRA expects in inspection. Pulled from file-13 Y2-Y3 to Phase 5 to align with the audit-workspace launch.

NFRA disciplinary context: **94 cumulative orders, 19 firms + 84 CAs penalised** ([TaxGuru on NFRA](https://taxguru.in/company-law/action-nfra-prevent-audit-failure-regulatory-lapses.html)); **Big 6 own 67% of Nifty 500 audits** ([PrimeDatabase](http://www.primedatabasegroup.com/newsroom/BIG_6_CONTINUE_DOMINANCE_OF_INDIAN_AUDIT_LANDSCAPE_2024-25.pdf)). The audit-firm tooling demand is real and growing.

### 14. MSME 45-day watchlist as a first-class entity

Sec 43B(h) compliance is a regulatory wedge. The watchlist needs a surfaced UI (UC3 in the use-cases catalogue), not buried in vendor master metadata. A dedicated `msme_watchlist` table with auto-population from vendor master (`vendors.is_msme=true`) and 45-day clock from invoice date.

### 15. ESG / BRSR Core taxonomy module

The SEBI BRSR Core glide path mandates: **150 (FY23-24) → 250 (FY24-25) → 500 (FY25-26) → 1,000 (FY26-27)** ([SEBI BRSR circular](https://www.sebi.gov.in/legal/circulars/jul-2023/brsr-core-framework-for-assurance-and-esg-disclosures-for-value-chain_73854.html), [KPMG firstnotes](https://assets.kpmg.com/content/dam/kpmgsites/in/pdf/2025/01/firstnotes-sebi-introduces-certain-key-changes-in-brsr-reporting.pdf.coredownload.inline.pdf)). CAOS is one customisation away from being a BRSR Core data-collection platform. Phase 6 module addition.

### 16. eMudhra / NSDL DSC integration

eMudhra is the largest DSC CA in India: **~40% market share, ₹527.8 cr FY25 revenue, 3.08M DSCs issued in FY24** ([ICRA rationale](https://www.icra.in/Rating/GetRationalReportFilePdf?id=128000)). Capricorn / Sify / Verasys / (n)Code / NSDL Protean / NIC / CDAC / IDRBT round out the ecosystem ([99digitalsignature](https://www.99digitalsignature.com/digital-signature-certificate-authority-in-india.php)). DSC integration is needed for in-house DSC management at scale (UC15, UC19, UC20). Phase 6 reseller integration.

### 17. Section 206C(1H) removed from product spec (immediate)

Was use case #8 in the original file-13 catalogue. **Removed in Budget 2025 effective 1 April 2025** ([TaxGuru FAQs](https://taxguru.in/income-tax/tcs-sale-goods-removed-april-1-2025-faqs.html)). Sec 194Q remains active and continues to be tracked. **Footnote already applied in [/caos/product/use-cases/](/caos/product/use-cases/) UC8.**

Both earlier work and the verified-numbers atlas reinforce: enterprise buyers will not adopt without integration paths. ## Tenant isolation summary (operationalising design decision)

The tenant isolation model is the single most architecturally consequential decision in the CAOS backend. The model is:

```
SHARED (default for new firms)
 └── RLS in `public` schema; SET LOCAL app.current_tenant per request
SCHEMA (paid tier; sysadmin upgrade)
 └── per-firm Postgres schema (firm_<slug>); search_path dispatched per request
DEDICATED (reserved; not implemented)
 └── per-firm DB host (firms.db_host); pgxpool keyed by host
```

Key invariants:

- Every query runs inside `WithTenant(ctx, firmID, isolationMode, schemaName, fn)`. Never `db.Pool.Query` directly outside system-admin code.
- Both SHARED and SCHEMA modes must work for every repo function. CI tests both modes for every test.
- DEDICATED has a working stub (per pre-investment #9) — clear "deferred" path documented.
- Migrations are written ONCE for `public` shape; the runner replays them per-schema verbatim for SCHEMA-mode firms.

The model is what allows the same code to serve a CA solo-prop on SHARED + RLS, a mid-tier audit firm on SCHEMA + per-firm Postgres schema, and (eventually) a BFSI / PSU enterprise on DEDICATED + VPC-isolated host. Three modes, one codebase.

## Money in INR paise (and the FX reservation)

Money is stored as **INR paise** (`int64`) in Tally voucher tables and across all money-bearing CAOS tables. Display is `₹` with Indian comma grouping (1,23,456.78). The `currency` column reservation (pre-investment #3) means that future FX support requires only:

- Populate `currency` with the appropriate ISO 4217 code on each row.
- Add a rates module (likely consuming RBI reference rates).
- Add revaluation jobs that consume the rates module and recompute INR-equivalents at period-end.

None of these requires schema changes on the money-bearing tables themselves. That is what reserving the currency column buys.

## Person decoupling — the multi-org membership consequence

The `persons` table is global (no `tenant_id`). It holds the universal identity (phone, email, name, OTP verification flags). Membership in firms and clients is via separate tables:

- `firm_members(tenant_id, person_id, role, status)` — person's role in a firm.
- `client_contacts(tenant_id, client_id, person_id, role,...)` — person's role with a client.

A single Person can be a `firm_member` of firm A (as PARTNER), a `client_contact` for client C in firm B (as DIRECTOR), and a `firm_member` of firm D (as ASSOCIATE) — all simultaneously, with the person's authentication identity unified. The shell switches based on active firm + active relationship.

This is the architectural commitment that makes the [in-house CA insight](/caos/stakeholders/composite/) tractable: the same Person record represents the CA who works at PwC (`firm_member` of PwC firm) AND who is also a director-contact for their personal HUF (CA's own business client at a different firm). One identity; many roles.

## entity_group for multi-entity (schema-only initially)

Per pre-investment #10, the `entity_group` table exists at launch with parent-subsidiary linkage. CA mode never surfaces it. The schema looks like:

```
entity_group (
 id UUID PK,
 tenant_id UUID NOT NULL,
 parent_client_id UUID NOT NULL REFERENCES clients(id),
 child_client_id UUID NOT NULL REFERENCES clients(id),
 ownership_pct NUMERIC(5,2) NOT NULL DEFAULT 100.00,
 effective_from DATE NOT NULL,
 effective_to DATE,
 created_at TIMESTAMPTZ DEFAULT NOW);
```

When a future Business or Enterprise SKU customer needs multi-entity consolidation context (UC13 — intercompany reconciliation, UC18 — multi-plant BRSR, UC25 — group Schedule III), the schema is already there. The UI ships at that point. The data does not need to be migrated.

## What is deliberately kept open vs. what is locked

The architecture distinguishes between three categories:

- **Locked.** Cannot be changed without an design decisions supersession. Examples: design decision (LLM provider interface), design decision (tenant isolation modes), design decision (session persistence), design decision (OTP-only — pending Year-2 successor).
- **Reserved (pre-investment).** Schema or interface is in place; functionality is deferred. Examples: `entity_group` (item 10), `currency` column (item 3), DEDICATED tenant stub (item 9), namespace reservations (item 4).
- **Open.** Not yet decided; will be decided when triggered. Examples: which OSS LLM model goes in `internal/provider/local-oss/`, what SLA tier sits on top of DEDICATED tenant when first BFSI customer asks, which specific eMudhra reseller agreement structure works best.

The discipline: *commit to the architectural surface; defer the business / packaging / pricing decisions.* This lets the engineering team move forward with high confidence on the substrate, while leaving SKU-level decisions for when actual customer evidence arrives.
